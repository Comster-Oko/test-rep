## 1. 要件定義で収集すべき情報

### 1.1 ビジネス要件

| 項目 | 内容 | 確認ポイント |
|------|------|------------|
| API の目的・背景 | なぜこの API が必要か | ユースケース、解決する課題 |
| 提供機能の概要 | APIで実現すること | 機能スコープの範囲 |
| 利用者・クライアント | 誰がこのAPIを使うか | 内部システム / 外部 / モバイル / Web |
| ビジネスルール | 業務上の制約や条件 | 業務フロー、承認フロー |
| 優先度・リリース時期 | いつまでに何が必要か | MVP範囲、フェーズ分け |

---

### 1.2 機能要件

#### エンドポイント定義

- 提供するリソース（例：ユーザー、注文、商品 など）
- 各リソースに対する操作（CRUD）
- エンドポイントの URL 設計方針（RESTful 原則に従うか）
- API バージョニング方針（例：`/api/v1/`）

#### リクエスト / レスポンス仕様

- リクエスト：HTTPメソッド、パスパラメータ、クエリパラメータ、リクエストボディ
- レスポンス：HTTPステータスコード、レスポンスボディの構造、フィールド定義
- エラーレスポンスの形式（共通フォーマットの有無）

#### データ要件

- 扱うデータの種類と属性（型、桁数、必須/任意）
- データの入力元と出力先
- 既存データベース・テーブルとの関連
- マスターデータ・コードの定義

---

### 1.3 非機能要件

| カテゴリ | 確認事項 | 具体例 |
|---------|---------|-------|
| **パフォーマンス** | 応答時間の目標値 | 95パーセンタイルで500ms以内 |
| **スループット** | 1秒あたりのリクエスト数 | ピーク時 1,000 RPS |
| **可用性** | 稼働率の目標 | 99.9%（月間ダウンタイム 44分以内） |
| **スケーラビリティ** | 負荷増加への対応方針 | 水平スケール対応の要否 |
| **セキュリティ** | 認証・認可方式 | JWT / OAuth2 / APIキー |
| **データ保護** | 個人情報・機密データの扱い | 暗号化、マスキング要否 |
| **ログ・監査** | ログ出力の要件 | アクセスログ、操作ログ |
| **国際化** | 多言語・タイムゾーン対応 | i18n対応要否、UTC統一か |

---

### 1.4 制約・前提条件

- 利用する Spring Boot バージョン
- JDK バージョン
- 連携する外部システム・API（依存関係）
- データベースの種類・バージョン（MySQL、PostgreSQL など）
- デプロイ環境（オンプレ / AWS / GCP / Azure / コンテナ）
- 既存システムとの互換性要件
- 法規制・コンプライアンス要件（個人情報保護法、GDPR など）

---

### 1.5 運用・保守要件

- 監視・アラートの要件
- バックアップ・リストア方針
- デプロイ方式（Blue-Green、ローリング など）
- SLA・サポート時間帯
- ドキュメント管理・更新ルール

## 2. 設計書に含めるべき内容

### 2.1 API 基本設計

#### ベース URL・バージョン管理

```
https://{host}/api/v1/{resource}
```

- バージョニング方式の選定（URLパス方式 / ヘッダー方式）
- 環境別エンドポイント（dev / stg / prod）の整理

#### 命名規則

| 対象 | 規則 | 例 |
|------|------|---|
| URL | スネークケース or ケバブケース（小文字） | `/user-profiles` |
| JSONフィールド | キャメルケース | `userId`, `createdAt` |
| HTTPメソッド | 操作の意味に沿った選択 | GET=取得、POST=作成、PUT=全更新、PATCH=部分更新、DELETE=削除 |

---

### 2.2 エンドポイント一覧

| No | エンドポイント | メソッド | 概要 | 認証要否 | 備考 |
|----|--------------|---------|------|---------|------|
| 1 | `/api/v1/users` | GET | ユーザー一覧取得 | 要 | ページネーション対応 |
| 2 | `/api/v1/users/{id}` | GET | ユーザー詳細取得 | 要 | |
| 3 | `/api/v1/users` | POST | ユーザー作成 | 要 | |
| 4 | `/api/v1/users/{id}` | PUT | ユーザー更新 | 要 | |
| 5 | `/api/v1/users/{id}` | DELETE | ユーザー削除 | 要 | 論理削除 |

> ※ 上記はサンプル。実際のAPIに合わせて記載する。

---

### 2.3 リクエスト / レスポンス定義

#### リクエスト定義例

```
POST /api/v1/users
Content-Type: application/json
Authorization: Bearer {token}
```

**リクエストボディ**

| フィールド名 | 型 | 必須 | 桁数・制約 | 説明 |
|------------|-----|------|-----------|------|
| `name` | String | ○ | 1〜100文字 | ユーザー名 |
| `email` | String | ○ | RFC5322準拠 | メールアドレス |
| `birthDate` | String | - | yyyy-MM-dd形式 | 生年月日 |
| `roleId` | Integer | ○ | 1〜9 | ロールID |

#### レスポンス定義例

**成功時（201 Created）**

```json
{
  "userId": 12345,
  "name": "山田 太郎",
  "email": "taro.yamada@example.com",
  "createdAt": "2025-04-01T09:00:00+09:00"
}
```

---

### 2.4 共通エラーレスポンス設計

```json
{
  "status": 400,
  "errorCode": "VALIDATION_ERROR",
  "message": "入力値に誤りがあります。",
  "details": [
    {
      "field": "email",
      "message": "有効なメールアドレスを入力してください。"
    }
  ],
  "timestamp": "2025-04-01T09:00:00+09:00",
  "traceId": "abc123xyz"
}
```

**HTTPステータスコード定義**

| ステータス | 用途 |
|-----------|------|
| 200 OK | 取得・更新成功 |
| 201 Created | 作成成功 |
| 204 No Content | 削除成功 |
| 400 Bad Request | バリデーションエラー |
| 401 Unauthorized | 認証エラー |
| 403 Forbidden | 認可エラー |
| 404 Not Found | リソース未存在 |
| 409 Conflict | 重複エラー |
| 422 Unprocessable Entity | ビジネスルール違反 |
| 500 Internal Server Error | サーバー内部エラー |

### 2.5 認証・認可設計

- 認証方式（JWT / OAuth2 / セッション / APIキー）
- トークンの発行・検証フロー（シーケンス図推奨）
- ロール・権限の定義とエンドポイントへの適用マッピング
- トークン有効期限とリフレッシュ方式
- Spring Security の設定方針（`SecurityFilterChain` の定義概要）

---

### 2.6 バリデーション設計

| バリデーション種別 | 実装箇所 | 使用アノテーション例 |
|-----------------|---------|------------------|
| 入力値検証（型・形式・必須） | Controller / DTO | `@NotNull`, `@Size`, `@Email`, `@Pattern` |
| ビジネスルール検証 | Service 層 | カスタムバリデーション |
| DB制約との整合性確認 | Repository / DB | ユニーク制約、外部キー制約 |

---

### 2.7 アーキテクチャ・レイヤー設計

```
[Client]
   │
   ▼
[Controller 層]     ← リクエスト受付、レスポンス返却、バリデーション
   │
   ▼
[Service 層]        ← ビジネスロジック、トランザクション管理
   │
   ▼
[Repository 層]     ← データアクセス（Spring Data JPA / MyBatis）
   │
   ▼
[Database]
```

- 各レイヤーの責務と依存方向を明確化
- DTO / Entity / Domain Model の変換箇所の定義
- 共通処理（例外ハンドリング、ログ、トレース）の実装箇所

---

### 2.8 例外ハンドリング設計

- `@ControllerAdvice` / `@RestControllerAdvice` による一元管理
- 業務例外・システム例外の分類
- 例外クラス階層の定義
- ログ出力レベルの定義（ERROR / WARN / INFO）

---

### 2.9 ログ設計

| ログ種別 | 出力内容 | 出力レベル |
|---------|---------|----------|
| アクセスログ | メソッド、URL、ステータス、処理時間 | INFO |
| エラーログ | スタックトレース、入力値（マスク済） | ERROR |
| 業務ログ | 重要な業務イベント | INFO / WARN |
| デバッグログ | 変数値、処理の流れ | DEBUG（本番無効） |

- ログフォーマットの統一（JSON形式推奨）
- 個人情報・機密情報のマスキング対象の定義
- `traceId` / `spanId` による分散トレーシング対応（必要な場合）

---

### 2.10 DB 設計との対応

- テーブル定義（ER図）との対応関係
- インデックス設計（パフォーマンスに影響するクエリの洗い出し）
- トランザクション境界の定義（`@Transactional` の適用箇所）
- 楽観的ロック / 悲観的ロックの使用箇所

---

### 2.11 パフォーマンス設計

- ページネーション方式（オフセット / カーソル）の採用根拠
- キャッシュ戦略（Spring Cache / Redis 等）の適用箇所
- N+1問題の対策（fetch join / バッチフェッチ）
- タイムアウト設定（Connection Timeout / Read Timeout）

---

### 2.12 API ドキュメント

- OpenAPI（Swagger）による仕様書の自動生成方針
- `springdoc-openapi` の導入・設定
- 各エンドポイントへの `@Operation` / `@Schema` アノテーション付与ルール

## 3. レビュー観点

### 3.1 要件定義レビュー観点

#### クライアント・発注者向け確認事項

- [ ] API の目的・提供価値がビジネス要件と一致しているか
- [ ] 機能スコープ（What to build）が明確に定義されているか
- [ ] 優先度・リリーススケジュールが合意されているか
- [ ] 非機能要件（性能・可用性）の目標値が妥当か

#### PL・PM 向け確認事項

- [ ] 全ユースケースが網羅されているか（漏れ・抜けがないか）
- [ ] 外部システム連携の依存関係が整理されているか
- [ ] 制約条件（技術・法規制）が全て洗い出されているか
- [ ] リスク（未確定事項・確認待ち）が明記されているか

---

### 3.2 設計書レビュー観点

#### API 設計の正確性

- [ ] RESTful 設計原則に従っているか（URLに動詞が含まれていないか）
- [ ] HTTPメソッドが操作の意味と一致しているか
- [ ] ステータスコードが適切に使い分けられているか
- [ ] エンドポイントの命名が一貫しているか

#### リクエスト・レスポンス設計

- [ ] 全フィールドに型・桁数・必須/任意が定義されているか
- [ ] レスポンスに不要な内部情報（スタックトレース、内部ID等）が含まれていないか
- [ ] 日時フィールドのフォーマットが統一されているか（ISO 8601推奨）
- [ ] ページネーションのレスポンス形式が適切か（totalCount、nextPageToken等）

#### セキュリティ

- [ ] 全エンドポイントの認証・認可要件が定義されているか
- [ ] 認証不要のエンドポイントが意図的なものか確認されているか
- [ ] OWASP API Security Top 10 の観点で問題がないか
  - Broken Object Level Authorization（BOLA）
  - 過剰なデータ露出（Excessive Data Exposure）
  - Mass Assignment の防止
- [ ] 個人情報・機密情報の取り扱いが適切か（暗号化、マスキング）
- [ ] レートリミットが検討されているか

#### バリデーション

- [ ] 全入力値にバリデーションが定義されているか
- [ ] バリデーションエラー時のレスポンスが詳細かつ適切か
- [ ] SQLインジェクション・XSS 対策が考慮されているか

#### 例外・エラーハンドリング

- [ ] 想定される全エラーケースが定義されているか
- [ ] エラーレスポンスのフォーマットが全エンドポイントで統一されているか
- [ ] エラーコードが一意で管理されているか
- [ ] システムエラー時にスタックトレースが外部露出しないか

#### パフォーマンス

- [ ] 大量データ取得時のページネーションが設計されているか
- [ ] N+1問題が発生しうる箇所が洗い出されているか
- [ ] キャッシュ戦略が検討されているか
- [ ] インデックスが適切に設計されているか

#### 設計の一貫性・保守性

- [ ] 命名規則がドキュメント全体で統一されているか
- [ ] レイヤー間の責務分担が明確か
- [ ] トランザクション境界が適切に定義されているか
- [ ] 将来の拡張性が考慮されているか（追加フィールド、バージョンアップ）

---

### 3.3 実装レビュー観点（コードレビュー）

#### Spring Boot 実装

- [ ] アノテーションの使い方が適切か（`@RestController`, `@Service`, `@Repository` 等）
- [ ] `@Transactional` の適用箇所・伝播設定が正しいか
- [ ] `@Valid` / `@Validated` によるバリデーションが適切に機能しているか
- [ ] DTO と Entity の変換が Service 層で行われているか（Controller でEntityを直接返していないか）
- [ ] `@RestControllerAdvice` で例外が一元管理されているか

#### セキュリティ実装

- [ ] Spring Security の設定が要件通りか
- [ ] パスワードが `BCryptPasswordEncoder` 等でハッシュ化されているか
- [ ] JWT の署名検証・有効期限チェックが実装されているか
- [ ] CORS 設定が適切か（必要最小限のオリジンのみ許可）
- [ ] SQL が JPA / Prepared Statement で構築されているか（動的クエリの安全性）

#### コード品質

- [ ] メソッドの責務が単一になっているか（単一責任原則）
- [ ] マジックナンバー・マジック文字列が定数化されているか
- [ ] ログに個人情報・機密情報が出力されていないか
- [ ] リソース（DB接続、ファイル等）が適切にクローズされているか
- [ ] 非同期処理・スレッドセーフが考慮されているか

#### テスト

- [ ] 単体テスト（Service 層）が実装されているか
- [ ] Controller 層のテスト（`@WebMvcTest`）が実装されているか
- [ ] 境界値・異常系のテストケースが網羅されているか
- [ ] テストデータが本番データを使用していないか
